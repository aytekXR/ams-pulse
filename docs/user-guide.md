# Pulse — User Guide

This guide walks through every screen in the Pulse web UI in sidebar order.
It is written against the shipped interface; stubs and future features are
noted explicitly.

> **Screenshots** are generated by running
> `node qa/marketplace/capture-live-screenshots.mjs` against a live instance.
> Reference images live under `marketplace/screenshots/` (e.g.
> `marketplace/screenshots/ss1-dashboard.png`). Do not treat absent PNGs as a
> build failure — regenerate them after any significant UI change.

## 1. Signing in

**URL:** `/` (redirects to token gate when unauthenticated)

Pulse protects every page behind an authentication gate. The gate is a
full-page centered card that renders before the sidebar loads.

![Sign-in screen](marketplace/screenshots/ug-login.png "Sign-in form
with an API Token field and a Sign in button")

### Token sign-in

Enter an API token in the **API Token** field (placeholder `plt_…`) and click
**Sign in**. The token is stored in the browser's `localStorage` for the
session. On a fresh install the first token is printed to the server's
standard error at first run — look for a line like:

```
pulse: FIRST RUN — generated admin token: plt_<48 hex chars>
       Save this token; it will not be shown again.
```

Additional tokens can be created and copied from **Settings → API Tokens**
(see [Settings](#13-settings)).

If the token is revoked or expires while you are logged in, all API calls
return HTTP 401. Pulse detects this, clears the stored token, and returns you
to the sign-in screen with the message:
_"Session expired or token revoked. Please enter your token again."_

### SSO sign-in

When the server is configured with an OIDC provider (environment variable
`PULSE_OIDC_*`), a **Sign in with SSO** button appears below a horizontal
divider. Clicking it redirects to `/auth/oidc/login` and then back to the
dashboard on success. Cookie-authenticated OIDC sessions are detected
automatically on page load — no token entry is needed.

Sign-out (the arrow icon in the sidebar footer) revokes the OIDC cookie
server-side via `POST /auth/oidc/logout` and then reloads the page.

## 2. First-run onboarding wizard

**URL:** `/onboarding`

New deployments without a configured AMS source are automatically redirected
here from the dashboard on first login. The redirect fires at most once per
browser: after the wizard is completed or skipped, a flag is written to
`localStorage` and the redirect never fires again. You can always reach the
wizard by navigating directly to `/onboarding`.

![Onboarding wizard step indicator](marketplace/screenshots/ug-onboarding-step2.png
"Four-step progress indicator: Welcome, Add source, Verify, Done")

The wizard has four steps shown in a numbered progress indicator at the top of
the page:

| Step | Label | What happens |
|------|-------|--------------|
| 1 | **Welcome** | Introduction card — lists what the wizard will help you do; click **Get started** to proceed |
| 2 | **Add source** | Form to configure an AMS REST endpoint (name, REST URL, optional username, credential env var, log path); click **Add source** to save |
| 3 | **Verify** | Fires `POST /admin/sources/{id}/test`; shows connection latency and AMS version on success, or the error message on failure; click **Continue** to proceed regardless |
| 4 | **Done** | Confirmation card with a checkmark; click **Go to live dashboard** |

A **Skip setup** link at the bottom of every step dismisses the wizard without
adding a source. If AMS was configured via the `PULSE_AMS_URL` environment
variable rather than the UI, the wizard is suppressed automatically (the
health check at `/healthz` reports `ams_env_configured: true`).

## 3. Live dashboard

**URL:** `/` (sidebar: **Live**)

![Live dashboard](marketplace/screenshots/ss1-dashboard.png "Stat cards,
protocol donut chart, by-application table, and active streams table")

The live dashboard is the default landing page after sign-in. It shows the
current state of your AMS cluster, updated in real time.

### Connection indicator

The top-right header shows either **Live** (green dot) or **Polling** (amber
dot):

- **Live** — a WebSocket connection to `/api/v1/live/ws` is open. The
  overview stats are pushed as `snapshot` and `delta` messages with no
  polling overhead.
- **Polling** — WebSocket is unavailable (proxy, firewall, or server not
  yet upgraded). Pulse falls back to polling the REST API every 5 seconds.
  All data is identical; only the latency differs.

### Stat cards

Four summary cards display cluster-wide totals pulled from
`GET /api/v1/live/overview`:

| Card | Metric |
|------|--------|
| **Viewers** | `total_viewers` — concurrent viewer count |
| **Publishers** | `total_publishers` — active ingest streams |
| **Avg CPU** | Mean `cpu_pct` across all discovered nodes |
| **Avg RAM** | Mean `mem_pct` across all discovered nodes |
| **Streams** | Count of active streams from the stream list |

CPU and RAM cards are hidden when no node data is available.

### Protocol mix donut

The donut chart to the left of the application table shows the `protocol_mix`
breakdown: **WebRTC**, **HLS**, **RTMP**, **DASH**, and **Other**. Hover a
segment to see the exact count.

### By application table

Lists each AMS application with its viewer count and publisher count. Empty
when no streams are active.

### Active streams table

Shows up to 500 active streams from `GET /api/v1/live/streams`, including
stream ID, application, state, viewer count, and ingest bitrate. The table
is virtualized; large fleets render without layout thrashing.

The **Refresh** button in the page header forces a new REST fetch regardless
of WebSocket state.

## 4. Analytics

**URL:** `/analytics` (sidebar: **Analytics**)

Historical audience data backed by ClickHouse. The date range defaults to the
last 7 days; the picker lets you choose any custom range.

![Analytics page](marketplace/screenshots/ss4-analytics.png "Audience
timeseries chart and summary stat cards")

### Tabs

| Tab | Content |
|-----|---------|
| **Audience** | Summary stat cards (total views, unique viewers, watch time, peak concurrency) plus a line chart of views / uniques / peak concurrency over time |
| **Geo** | Table: country, views, unique viewers, watch time |
| **Device** | Table: device type, browser, OS, views |

### Date range

Select the start and end date from the date range picker above the tabs.
Changing the range refetches all three tab datasets simultaneously.

### Export CSV

The **Export CSV** button in the top-right triggers a browser download of the
current audience data for the selected date range via
`GET /api/v1/analytics/audience?format=csv`.

## 5. QoE

**URL:** `/qoe` (sidebar: **QoE**)

**Tier required: Pro or higher.**

Viewer Quality of Experience data collected by the
[beacon SDK](beacon-sdk.md). Install the beacon in your player to start
populating this screen.

![QoE page](marketplace/screenshots/ug-qoe.png "QoE summary cards and
bitrate timeline chart")

### Summary cards

| Metric | Meaning |
|--------|---------|
| **Startup p50** | Median time from player load to first frame, in milliseconds — the experience for a typical viewer |
| **Startup p95** | 95th-percentile startup time — the experience for viewers with slow connections or cold caches |
| **Rebuffer Ratio** | Fraction of playback time spent stalled waiting for data; values above 5% are flagged HIGH |
| **Error Rate** | Fraction of sessions that encountered a playback error; values above 1% are flagged HIGH |

### Bitrate timeline

Line chart showing bitrate p50 and p95 (kbps) over the selected time window.
Both series are animated-off at all times (charts never slide in, per the
design system motion policy).

### Filters

The **Stream ID filter** and **App filter** fields narrow data to a specific
stream or AMS application. Both are plain text; leave blank to see all traffic.

### Empty state

If no beacon data has been received yet, the page shows:
_"Viewer QoE data is collected by the beacon SDK."_
with a link to the [SDK setup docs](beacon-sdk.md). No data appears until at
least one player with the beacon is active.

## 6. Ingest health

**URL:** `/ingest` (sidebar: **Ingest**)

Per-publisher ingest quality for the last 15 minutes. Refreshes automatically
every 15 seconds; a manual **Refresh** button is also provided.

![Ingest health page](marketplace/screenshots/ss2-ingest-health.png "Publisher
list with health score badges and a detail panel")

### Publisher list

Each active publisher appears as a row with:

- **Stream** — stream ID in monospace
- **App** — AMS application name
- **Node** — node ID where the publisher is ingest (or `—` for single-node)
- **Health** — a filled progress bar and a status badge:
  - `Healthy` (green) — score ≥ 80
  - `Degraded` (amber) — score 50–79
  - `Poor` (red) — score < 50
- **Drops** — count of drop events in the window, shown in red if non-zero

### Health score

The score (0–100) is a weighted composite of bitrate, FPS, keyframe interval,
packet loss, and jitter. The full formula is documented in
[docs/ARCHITECTURE.md §10](ARCHITECTURE.md#10-ingest-health-score-formula-f4).

### Detail panel

Click any row (or the **Details** button) to expand a detail panel below the
table showing:

- Drop event timestamps and reasons, highlighted in red
- **Bitrate & FPS** timeseries chart with drop-event reference lines
- **Packet Loss & Jitter** chart with a 1% packet-loss warning threshold line

Click **Collapse** or **Close** to dismiss the panel.

## 7. Alerts

**URL:** `/alerts` (sidebar: **Alerts**)

Rule-based alerting with three tabs: **Rules**, **Channels**, and **History**.

![Alerts page](marketplace/screenshots/ss3-alerting.png "Alert rules list
with severity badges and edit controls")

### Rules tab

Lists all configured alert rules. Each rule row shows the rule name, metric
expression (`metric operator threshold · window · cooldown`), severity badge,
and enabled/muted state.

**Creating a rule** — click **+ New rule** to open the inline form:

| Field | Notes |
|-------|-------|
| Rule type | `threshold` (metric vs fixed value) or `anomaly` (statistical deviation) |
| Name | Required; used as the display label throughout the UI |
| Metric | e.g. `viewer_count`, `cpu_pct`, `ingest_bitrate_kbps`, `node_down` |
| Operator / Threshold | For threshold rules: `gt`, `lt`, `gte`, `lte`, `eq` with a numeric value |
| Sigma / Min Samples | For anomaly rules: sigma replaces operator+threshold; window is locked to 3600 s |
| Severity | `info`, `warning`, or `critical` |
| Window | Evaluation window in seconds (60 s – 3600 s) |
| Cooldown | Minimum seconds between repeat notifications |
| Scope | Optional stream ID / app / node ID filter; leave blank to match all |
| Enabled | Uncheck to pause the rule without deleting it |
| Muted | Rule is evaluated and recorded but sends no notifications |

**Default "Stream offline" rule** — fresh installs include a built-in rule for
stream disconnection events. This rule ships **muted** so it does not flood
channels on the first run. Enable notifications by editing the rule and
unchecking **Muted**.

**Maintenance windows** — use the **Muted** toggle on any rule to suppress
notifications during planned maintenance. Unmute the rule when maintenance
ends; history entries are still recorded while muted.

Deleting a rule shows an inline confirmation banner before the API call is
made.

### Channels tab

Notification destinations. Supported types and the tier they require:

| Type | Tier |
|------|------|
| Email | Free |
| Slack | Pro |
| Telegram | Pro |
| PagerDuty | Business |
| Webhook | Business |

Click **+ New channel** to configure a destination (webhook URL, email
address, Slack incoming-webhook URL, etc.). The **Test fire** button sends a
test payload immediately and reports accepted / failed with the channel's
response message.

Deleting a channel requires confirmation; existing rules that route to the
deleted channel stop notifying.

### History tab

Append-only log of every fired alert: rule ID, severity, state (`firing` or
`resolved`), timestamp, and the metric value that triggered the event. The
last 100 entries are shown (display cap; the server stores up to 1,000
history rows per rule before pruning oldest-first).

## 8. Reports

**URL:** `/reports` (sidebar: **Reports**)

**Tier required: Business or higher.**

On Free or Pro plans the page shows an upsell gate:
_"Usage Reports requires Business tier"_.

![Reports page](marketplace/screenshots/ss5-reports.png "Usage table with
viewer-minutes and egress columns")

### Usage tab

Tabular usage data for a chosen date range:

| Column | Notes |
|--------|-------|
| App | AMS application |
| Stream | Stream ID (monospace) |
| Tenant | Tenant label if matched; `—` if unassigned |
| Viewer-Min | Total viewer-minutes |
| Peak | Peak concurrent viewers |
| Egress GB * | Estimated egress; hover the `*` for the estimation method |
| Recording GB | Recorded bytes |

Totals appear as summary cards above the table. The **Export CSV** button
downloads the current range as a CSV file via
`GET /api/v1/reports/export?format=csv`.

### Schedules tab

Automated report delivery. Each schedule specifies:

- **Frequency** — a cron preset (monthly on the 1st at 06:00 UTC, weekly
  Monday 06:00 UTC, daily 06:00 UTC) or a custom cron expression in UTC
- **Format** — `CSV` or `PDF` (select per schedule; the format badge shows
  `CSV` or `PDF` in the schedule list)
- **App scope / Tenant scope** — optional filters; leave blank for all traffic

Create schedules with **+ New schedule**; edit or delete from the schedule
list.

### Tenants tab

Defines stream-to-billing-entity mappings used by the usage table's **Tenant**
column. Each tenant matches streams by:

- **Stream pattern** — SQL `LIKE` or regex applied to `stream_id`
- **Meta tag key + value** — beacon metadata key/value pair

At least one matcher is required. Unmatched streams appear as `—` in reports.

## 9. Fleet

**URL:** `/fleet` (sidebar: **Fleet**)

Cluster node inventory. Auto-discovers nodes within 2 minutes of first
contact; refreshes automatically every 30 seconds.

![Fleet page](marketplace/screenshots/ug-fleet.png "Node cards with CPU
and memory load bars")

### Summary header

When nodes are present, five aggregate tiles show: **Total Nodes**, **Up**,
**Degraded**, **Origins**, and **Edges**.

### Roles and status

Each node carries a **role** badge (`origin` or `edge`) and a **status** badge
(`up`, `degraded`, or `down`). Origin and edge nodes with the same `node_id`
are de-duplicated.

### View toggle

Switch between **Cards** (grid of node cards) and **Table** (dense row per
node) using the segmented control in the header. Both views show:

- Node ID (monospace)
- AMS version
- Last-seen timestamp
- CPU load bar — green below 60%, amber 60–80%, red above 80%
- Memory load bar — blue (normal range), amber above 70%, red above 85%
- Network in/out (Mbps)

## 10. Anomalies

**URL:** `/anomalies` (sidebar: **Anomalies**)

**Tier required: Enterprise.**

On lower plans the page shows an upsell gate:
_"Anomaly Detection requires Enterprise tier"_.

Statistical anomaly flags generated by the Welford online detector. The
detector computes a rolling mean and standard deviation for each metric;
when an observation deviates by more than N standard deviations (sigma), a
flag is raised.

![Anomalies page](marketplace/screenshots/ug-anomalies.png "Anomaly flags
table with sigma values and severity badges")

### Columns

| Column | Meaning |
|--------|---------|
| Metric | The metric that deviated (e.g. `cpu_pct`, `viewer_count`) |
| Scope | node / app / stream the deviation was detected on |
| Observed / Expected | The actual value and the baseline mean |
| Delta | Signed difference (observed minus expected) |
| Sigma | Number of standard deviations from the mean; higher sigma = more unusual |
| Severity | Low (σ ≥ 2), Medium (σ ≥ 3), High (σ ≥ 4) |
| Detected at | Timestamp of the anomalous sample |

### Sensitivity selector

The **Sensitivity** dropdown sets the minimum sigma for flags to appear:
_All (σ ≥ 2)_, _Medium+ (σ ≥ 3)_, or _High (σ ≥ 4)_. Lower sigma means more
flags; raise the threshold to reduce noise.

### Empty state — baselines still learning

When no flags appear, the empty state reads:
_"Baselines are still learning — anomaly flags appear once enough samples have
been collected (typically a few hours of traffic). No action needed while
baselines are building."_

This is expected on a fresh deployment or after a restart. Flags will appear
automatically once the detector has enough samples.

## 11. Probes

**URL:** `/probes` (sidebar: **Probes**)

**Tier required: Pro or higher.**

On Free plans the page shows an upsell gate:
_"Synthetic Probes requires Pro tier"_.

Synthetic probes periodically fetch a stream URL from outside your
infrastructure and record Time to First Byte (TTFB), measured bitrate, and
success/failure. All probe results are clearly labeled **Synthetic** — they are
never silently mixed into organic QoE charts.

![Probes page](marketplace/screenshots/ss6-probes.png "Probes list with
last result and Results panel")

### Probe list

Each probe row shows: name, URL, protocol badge, check interval, enabled/off
badge, and the last result's status + TTFB.

### Protocols

Each probe targets one protocol:

| Protocol | Notes |
|----------|-------|
| HLS | Fetches the playlist and first segment |
| DASH | Fetches the manifest |
| RTMP | Connects and measures time to first keyframe |
| WebRTC | Full ICE/DTLS handshake; records signaling and ICE state |

### Creating a probe

Click **+ New Probe** and fill in:

- **Name** — display label
- **Stream URL** — full URL including scheme (`https://`, `rtmp://`, etc.)
- **Protocol** — must match the URL
- **Interval** — check frequency in seconds; minimum 30 s
- **Timeout** — per-check timeout in seconds; minimum 1 s
- **Enabled** — uncheck to create the probe without starting runs immediately

### Results panel

Click **Results** on any probe row to expand a panel below the table. The
panel header always shows the **Synthetic** badge and the notice
_"not organic viewer data"_.

The panel contains:
- TTFB timeline chart (p50 TTFB and segment TTFB, with an amber 500 ms
  reference line)
- Bitrate timeline chart (when bitrate data is available)
- Recent results table (up to 20 rows) with columns: Time, Type (Synthetic
  badge on every row), Status, TTFB, Segment TTFB, Bitrate, Signaling, Connect
  time, ICE state, RTT, Jitter, Loss, Error

## 12. Audit log

**URL:** `/audit-log` (sidebar: **Audit Log**)

Read-only, append-only log of every mutating API call made to admin and
configuration endpoints. No tier gate — all authenticated admin users can view
the log.

![Audit log](marketplace/screenshots/ug-audit-log.png "Audit log table
with Time, Actor, Action, Object, Object ID, Source IP columns")

### Columns

| Column | Notes |
|--------|-------|
| Time | Timestamp (local time) |
| Actor | Token display name, or `token <first-8-chars>…` when no name is set; hover for OIDC user ID |
| Action | API verb badge — e.g. `create`, `update`, `delete` |
| Object | Resource type — e.g. `alert_rule`, `source`, `token` |
| Object ID | ID of the affected resource; hover for JSON detail |
| Source IP | Remote address of the request |

Covered resources: alert rules and channels, users, API and ingest tokens,
probes, report schedules, AMS sources, tenants, and the license key.

### Pagination

The log is keyset-paginated newest-first. Click **Load more** at the bottom to
fetch the next 50 entries. Click **Refresh** to reload from the top.

## 13. Settings

**URL:** `/settings` (sidebar: **Settings**)

Six tabs accessible via the top tab bar (wraps on narrow screens):

| Tab | Purpose |
|-----|---------|
| Sources | Manage AMS source connections |
| API Tokens | Create and revoke dashboard/API bearer tokens |
| Ingest Tokens | Create and revoke beacon SDK tokens |
| Integrations | Prometheus scrape URL, S3 export destination |
| License | View current tier and activate a license key |
| Users | Stub — see below |

### Sources tab

Lists configured AMS sources with their type badge and REST URL. Click
**+ Add source** to be directed to the onboarding wizard (the canonical path
for adding sources). Remove a source with the **Remove** button.

### API Tokens tab

Lists API tokens with their scope (`admin` or `read`), creation date, and last
used date.

Click **+ New token** and enter a name; a prompt asks whether the token should
have admin (write) scope or read-only scope. The generated token value is
shown once in a copyable inline block — it is hashed immediately by the server
and cannot be retrieved again.

### Ingest Tokens tab

Tokens that authenticate the beacon SDK. Click **+ New ingest token** and
enter a name. The created token and a ready-to-paste SDK snippet are shown
once:

```js
import { Pulse } from '@pulse/beacon';

const session = Pulse.init({
  ingestUrl: 'https://your-pulse-host', // the SDK appends /ingest/beacon itself
  token: 'plt_…',                       // the ingest token created above
  streamId: 'your-stream-id',           // must match the AMS stream id being played
});
```

See [docs/beacon-sdk.md](beacon-sdk.md) for full SDK integration instructions.

### Integrations tab

**Prometheus** — displays the scrape URL (`<host>/metrics`) and a sample
`scrape_configs` block for `prometheus.yml`. The endpoint is unauthenticated
by default; set `PULSE_METRICS_TOKEN` to require a bearer token.

**S3 Export Destination** — configure automatic report uploads. Enter the S3
bucket name, AWS region, and the names of environment variables that hold the
access key and secret key. Credential values are never stored — only the
environment variable names.

### License tab

Shows the current tier badge, expiry date (if any), and the limits grid
(e.g. `sources`, `streams`; `∞` means unlimited).

Enter a license key in the format `PULSE-XXXX-XXXX-XXXX` and click
**Activate** to upgrade the tier. Free tier requires no key.

The header area between the main header and page content shows a persistent
trial banner when the license is approaching expiry:

- **14 days or fewer remaining** — amber warning strip:
  _"License expires in N days — activate a key in Settings › License."_
  (dismissable for the browser session)
- **Expired** — red error strip (non-dismissable):
  _"License expired — Pulse is running on Free tier limits. Activate a
  license in Settings › License."_

### Users tab

User management is **API-only** in the current release. The UI tab shows a
placeholder: _"User management — coming in a future update."_

To manage users (create, list, deactivate) use the admin API directly. See
[docs/admin-guide.md](admin-guide.md) for the full API reference.

## Sidebar footer controls

The bottom of the left sidebar (below the navigation links) has two
preference controls that apply globally:

**Theme toggle** — switches between dark and light mode. The icon shows a sun
when in dark mode (click to switch to light) and a moon when in light mode
(click to switch to dark). The preference is persisted across page reloads.

**Density segmented control** — three options:

| Value | Effect |
|-------|--------|
| **Default** | Standard row heights and spacing |
| **Compact** | Reduced padding; fits more rows on screen |
| **Wall** | Minimal padding; suited for operator display boards |

The density and theme settings are stored in `localStorage`.
