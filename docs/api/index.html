Pulse API (1.0.0)

Download OpenAPI specification:

License: Proprietary

Pulse Query & Config API. Serves the web dashboard and the customer-facing Data API (PRD F8). All /api/v1 paths require bearer-token authentication unless noted. Read endpoints answer from pre-built rollups (ClickHouse) or live in-memory aggregates — never from raw event tables directly.

Live endpoints (/live/overview, /live/streams, /live/ws) serve from in-memory aggregates maintained by the collector, not from ClickHouse. This gives the sub-10 s latency required by PRD F1 while keeping ClickHouse load low.

Timestamps: All time parameters accept either Unix epoch milliseconds (integer) or RFC 3339 strings (e.g. 2026-01-01T00:00:00Z). Responses always return Unix epoch milliseconds integers.

Pagination: List endpoints that may grow large use cursor pagination via cursor (opaque string) and limit (max 500, default 50). A next_cursor field in the response envelope signals more pages; absent or null means last page.

Error envelope: All 4xx/5xx responses use the Error schema.

Data API parity: The public Data API (F8) exposes the same /api/v1 surface under a separate ingest token scope; no private endpoints exist.

WebSocket upgrade for live dashboard push

Upgrades to a WebSocket connection. The server pushes typed messages:

  • snapshot — full LiveOverview state, sent immediately on connect and periodically (default every 5 s).
  • delta — partial update object containing only changed fields; client merges into its local state.
  • heartbeat — {"type":"heartbeat","ts":<epoch_ms>} sent every 30 s.

Authentication: bearer token in Authorization header, token query parameter (WebSocket clients may not set headers), or the pulse_session cookie (OIDC/SSO browser sessions). All three are validated the same way as file downloads.

Message envelope schema:

{
  "type": "snapshot | delta | heartbeat",
  "ts": 1700000000000,
  "payload": { ... }
}

payload for snapshot/delta matches the LiveOverview schema (delta carries only changed keys). heartbeat has no payload.

string

Rule evaluation mode. threshold (default): classic operator+threshold comparison. anomaly: fires when the metric deviates beyond sigma standard deviations from its rolling Welford baseline (stored in anomaly_baselines). For anomaly rules, operator and threshold are ignored by the evaluator. Supported anomaly metrics are viewer_count, ingest_bitrate_kbps, cpu_pct, mem_pct, disk_pct, and ams_api_latency_ms; window_s must be 3600 (the Detector window). Unsupported metric or window value returns 400.

Default: 4

Anomaly rules: sigma multiplier for baseline deviation detection. Default: 4.0 (DefaultSigma).

SSO/OIDC login flow — phase 1 (server side)

Initiate OIDC authorization code flow

Redirects the browser to the configured OIDC provider's authorization endpoint. Requires PULSE_OIDC_ISSUER to be set; returns 501 otherwise. Sets a short-lived pulse_oidc_state cookie (HttpOnly, 10-minute max-age) containing the HMAC-signed state+nonce to prevent CSRF. Uses PKCE S256 to protect the authorization code exchange (code_challenge_method=S256). Phase 1: UI login button deferred to phase 2; this endpoint can be navigated to directly.

OIDC authorization code callback

Receives the authorization code from the OIDC provider, exchanges it for tokens, validates the id_token (iss/aud/exp/sig/nonce), maps groups to roles, creates a short-lived session token, and sets the pulse_session cookie (HttpOnly, SameSite=Lax). Then redirects to /. State parameter is verified against the pulse_oidc_state cookie to prevent CSRF.
