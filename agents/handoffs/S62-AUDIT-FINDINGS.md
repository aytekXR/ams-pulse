## [4] HIGH — Attacker-controlled printf format specifier in expandSegmentTemplate causes gigabyte string allocation
- **loc:** `server/internal/prober/probe_dash.go:372`  ·  **lens:** prober-parsing  ·  status: ✅ DONE (D-127, PR #124, prod v0.4.0-68-g2a122fd) · **NOTE:** positive allowlist `reSafeNumberSpec=^%0?\d{0,3}d$` (width ≤ 999) — only the DASH-defined `%0<width>d` printf form reaches fmt.Sprintf; anything else (e.g. `%999999999d`) degrades to plain `%d`. Mutation-proven. **Adversarial review found a sibling sink now ALSO fixed:** `$RepresentationID$` via `strings.ReplaceAll` was itself unbounded (count×len(id) → TB-scale within the 16 MiB body cap) — now bounded by `maxExpandedTemplateBytes=64 KiB` before expansion (mutation-proven separately). Width bound widened 2→3 digits (spec-legal %100d).
- **mechanism:** expandSegmentTemplate extracts the format specifier from the server's MPD SegmentTemplate media attribute using regex reNumberFmt (`\$Number%[^$]+\$`). The `[^$]+` group allows any sequence of non-`$` bytes after `%`, so an attacker can supply an arbitrary printf width. The extracted spec string is then passed verbatim to fmt.Sprintf(spec, number). A width such as `%999999999d` causes fmt.Sprintf to pad the integer to 999,999,999 characters. On 64-bit Go, parsenum does not overflow for 9-digit widths; the runtime attempts the full allocation.
- **scenario:** A malicious server returns a valid MPD containing SegmentTemplate media="seg-$Number%999999999d$.mp4". parseMPD reaches expandSegmentTemplate("seg-$Number%999999999d$.mp4", repID, 1). reNumberFmt matches `$Number%999999999d$`; spec becomes "%999999999d"; fmt.Sprintf("%999999999d", 1) attempts to allocate a ~1 GB string on the heap. If several probes hit different hostile servers concurrently, the combined allocation immediately exceeds available RAM. The malicious server needs to send only a few hundred bytes of MPD.
- **mutation/test:** Validate spec before calling Sprintf — e.g., reject any spec that does not match `^%0*\d{1,6}d$` (up to 6-digit width) or rewrite expandSegmentTemplate to do the zero-padding manually using strconv. A unit test calling expandSegmentTemplate("$Number%999999999d$", "r", 1) must return an error (or a bounded fallback string) rather than allocating the giant string; without the fix the test would allocate ~1 GB or be killed.
- **fix sketch:** Validate spec before calling fmt.Sprintf. Add a compiled regexp such as `var reValidNumberSpec = regexp.MustCompile(`^%0*[1-9]?\d{0,4}d$`)` (allowing at most a 5-digit zero-padded decimal width, max 99999 chars) and check it before the Sprintf call: if !reValidNumberSpec.MatchString(spec) { return fmt.Sprintf("%d", number) }. Alternatively, rewrite the substitution using strconv.FormatInt with manual zero-padding via strings.Repeat to eliminate fmt.Sprintf entirely from the hot path. The reversion test: expandSegmentTemplate("seg-$Number%999999999d$.mp4", "r", 1) must not allocate more than a bounded amount (e.g., assert len(result) < 1000) — without the fix this assertion fails (the result is a ~1 GB string or the test goroutine is OOM-killed).
- **verifier reasoning:** The vulnerability is real and fully traceable in the code. In expandSegmentTemplate (probe_dash.go:366-377), the regex reNumberFmt (`\$Number%[^$]+\$`) matches any token of the form `$Number%...$ ` where the `...` can be any sequence of non-`$` bytes. The spec is extracted verbatim at line 371: `spec := m[len("$Number"):len(m)-1]`. For the input `$Number%999999999d$` (length 19), this yields `spec = "%999999999d"` (m[7:18]). That spec is passed directly to `fmt.Sprintf(spec, number)` at line 372 with no width validation. Go's fmt package does not cap printf widths; it will attempt to allocate a buffer of approximately 999,999,999 bytes (~953 MiB) to satisfy the padding. The manifest body reaching parseMPD has no io.LimitReader guard (the LimitReader at line 196 applies only to the segment body), so an attacker-controlled server returns a normal-sized MPD (a few hundred bytes) and the oversized allocation is triggered purely from the media attribute value. No validation of the extracted spec exists anywhere in the call chain. A non-vacuous test: calling expandSegmentTemplate("seg-$Number%999999999d$.mp4", "r", 1) must return an error or a bounded fallback string; without the fix the test goroutine attempts a ~1 GB allocation. Severity HIGH is not overstated: this is a remote DoS requiring only that the attacker control any server configured as a probe target.
